# identity / professions

Reference data + future-import foundation for **Zarządzaj → Sekcja zawodowa**.

## Status truth

- **Categories:** `REFERENCE_DATA_READY` — 30 seeded categories, sorted by
  `order` ASC, status `active`, no PII.
- **Professions:** `DATA_PENDING` — no rows; the owner imports the full
  dataset later. `searchProfessions` returns `{ state: "data_pending" }`.
- **Specializations:** `DATA_PENDING` — same; `listSpecializations` returns
  `{ state: "data_pending" }`.
- **Import pipeline:** `IMPORT_CONTRACT_READY` / `DRY_RUN_ONLY` —
  `dryRunImport` validates a prospective package (empty name, slug shape,
  unknown category, duplicate slug/name) and reports would-be inserts;
  `report.persisted` is always `false`. No DB push.
- **Admin moderation:** `SKELETON_ONLY` — pure approve/reject/merge status
  logic + types, no UI, no store.
- **Custom specializations:** prepared (`CustomSpecializationDraft`), NOT
  enabled.

## What came from legacy

Source material only (from `ze starego/kontakty.zip` →
`drizzle/schema/identity/professions.ts`): the category model
(name/slug/icon/order), the 30 category names + icons, the
`profession_categories` / `professions` / `profile_professions` shape and the
`active/pending/rejected` status idea.

## What was NOT copied

Drizzle/Postgres/Supabase coupling, tRPC routers, legacy hooks, the
`getAll(200)` cap, and the `rzemiosto` slug typo (fixed to `rzemioslo`). The
legacy extract carried only the schema, not the category seed rows, so slugs
are generated by a Polish-aware slugify (`policy.normalizeSlug`).

## Boundaries

`server/domains-v2/identity/professions/**` imports only `@shared/contracts/*`
and sibling files. Public DTOs live in `@shared/contracts/professions`. The
frontend never imports `@server/*`; it consumes the shared contract via a
feature-local adapter (`MOCK_LOCAL_ONLY`).
