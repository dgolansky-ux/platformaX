# Notifications V2 — Persistence Migration Draft (Slice 14)

Status: **DRAFT_NOT_APPLIED** — `pnpm db push` was NOT executed; the
in-memory adapter at `server/domains-v2/notifications-v2/store.ts` continues
to back the runtime. This file is the migration draft that the durable
adapter will follow.

## Tables

### `notifications`

```sql
CREATE TABLE notifications (
  id                TEXT PRIMARY KEY,
  recipient_user_id TEXT NOT NULL,
  actor_user_id     TEXT NULL,
  type              TEXT NOT NULL,
  category          TEXT NOT NULL,
  title             TEXT NOT NULL,
  body_preview      TEXT NOT NULL,
  source_domain     TEXT NOT NULL,
  source_type       TEXT NOT NULL,
  source_id         TEXT NOT NULL,
  route_target      TEXT NOT NULL,
  status            TEXT NOT NULL,
  delivery_channel  TEXT NOT NULL DEFAULT 'in_app',
  dedupe_key        TEXT NULL,
  correlation_id    TEXT NULL,
  created_at        TIMESTAMPTZ NOT NULL,
  read_at           TIMESTAMPTZ NULL,
  archived_at       TIMESTAMPTZ NULL
);

CREATE INDEX notifications_recipient_status_created_idx
  ON notifications (recipient_user_id, status, created_at DESC);

CREATE INDEX notifications_recipient_category_created_idx
  ON notifications (recipient_user_id, category, created_at DESC);

CREATE UNIQUE INDEX notifications_dedupe_key_uidx
  ON notifications (recipient_user_id, dedupe_key)
  WHERE dedupe_key IS NOT NULL;
```

### `notification_settings`

```sql
CREATE TABLE notification_settings (
  user_id        TEXT NOT NULL,
  category       TEXT NOT NULL,
  in_app_enabled BOOLEAN NOT NULL DEFAULT TRUE,
  updated_at     TIMESTAMPTZ NOT NULL,
  PRIMARY KEY (user_id, category)
);
```

## Privacy notes

- `notifications.title` and `notifications.body_preview` carry **only**
  short, generic copy generated by the orchestrator. Never the raw post
  body, never the actor's email / phone, never the recipient's contact.
- `source_*` columns hold ids + a route target only.
- `dedupe_key` is the upstream event id; it stays in the column for
  idempotency but is **not** exposed on the public DTO.

## Migration safety

- Both tables are new; no destructive change on existing data.
- Indexes are designed to back the read-models the service exposes:
  * `listForViewer(filter=all|unread|category)`
  * `countUnreadByCategory`
  * `findByDedupeKey`
- `dedupe_key` unique partial index enforces idempotency at the DB level
  exactly the same way the in-memory adapter enforces it today.

## Rollout

1. Land this draft.
2. Add the durable adapter implementing `NotificationRepository` and
   `NotificationSettingsRepository` against this schema.
3. Switch the wiring in `application-v2` to the durable adapter behind a
   flag.
4. Backfill: not required — notifications are forward-only.

This draft is intentionally minimal. No backwards-compatibility shims, no
shadow tables, no extra columns "for the future".
