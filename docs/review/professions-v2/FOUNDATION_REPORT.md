# Sekcja zawodowa V2 — Foundation report

Status: `ACTIVE`
Branch: `feat/contacts-v2-clean-room-slice`
Scope: foundation only — 30 categories + clean V2 domain + import pipeline +
UI shell. Professions and specializations data are **DATA_PENDING** (the owner
ships the full package later).

## 1. Current scope (what was built)

- **30 profession categories** as clean V2 reference data, single source of
  truth in `shared/contracts/professions-categories.ts` (imported by both the
  server seed and the frontend adapter — no `@server/*` in the client, no
  drift). `id/name/slug/icon/order/status`, sorted by `order` ASC.
- **Domain module** `server/domains-v2/identity/professions/`:
  `dto.ts`, `contracts.ts`, `policy.ts`, `service.ts`, `repository.ts`
  (in-memory reference adapter), `mapper.ts`, `public-api.ts`,
  `seeds/profession-categories.seed.ts`, `import/professions-import.contract.ts`,
  `import/professions-import.validator.ts`, `moderation.ts`, `README.md`,
  `__tests__/`.
- **Import pipeline (dry-run / validate-only):** `dryRunImport` reports empty
  names, invalid slugs, unknown categories, duplicate slug/name, and would-be
  insert counts. `report.persisted` is always `false`. No DB push.
- **Frontend** `client/src/features-v2/identity/professional-section/` mounted
  at **Zarządzaj → Sekcja zawodowa** (`/manage/sekcja-zawodowa`, reachable from
  the desktop sidebar): title, lead, status pills, 30-category grid
  (selected/hover/focus, responsive 2→3→4 cols), step flow (kategoria →
  zawód DATA_PENDING → specjalizacja DATA_PENDING), "Moje zawody" slots (max 3,
  1 główny + 2 dodatkowe, disabled save), and a local-draft profession-proposal
  form (disabled submit).

## 2. What came from legacy

Source material only (from `ze starego/kontakty.zip` →
`drizzle/schema/identity/professions.ts`): the category model
(name/slug/icon/order), the 30 category names + icons, the
`professions` / `profile_professions` shape (max 3, primary/additional), the
`active/pending/rejected` status idea, and the proposal/merge concept.

## 3. What was rejected from legacy

Drizzle/Postgres/Supabase coupling, tRPC routers, legacy hooks, the
`getAll(200)` cap, the legacy custom-specialization `temp-*` id bug, and the
`rzemiosto` slug typo (fixed → `rzemioslo-i-uslugi-techniczne`). The extract
shipped only the schema (no category seed rows), so slugs are generated by a
Polish-aware slugify (`policy.normalizeSlug`); a future import package can
override them 1:1.

## 4. Future data import

The owner will send the full professions + specializations dataset. The
contract (`ImportProfessionRow`, `ImportDryRunReport`) and the validator are
ready; importing is **DRY_RUN_ONLY** today — nothing is persisted. When a
runtime store + transport land, the same validator gates the real import.

## 5. UI shell readiness

`UI_SHELL_ONLY` + `MOCK_LOCAL_ONLY`. Categories render from the shared
reference list via a feature-local adapter. No `@server/*` import. No fake
profession list, no fake save, no `window.alert/confirm`, no no-op buttons
(every disabled control is an honest "available after import").

## 6. Backend partial readiness

`BACKEND_PARTIAL`. Pure policy + in-memory reference adapter + service +
mapper + dry-run validator + moderation status logic, all unit-tested. No DB,
no transport.

## 7. Disabled / data-pending elements

- Profession selection (step 2) — DATA_PENDING.

## 8. What is intentionally NOT implemented

admin UI, real proposal save, custom-specialization save, profile embedding of
chosen professions (the profile professional section will consume this later —
`PROFILE_PROFESSIONAL_SECTION_PENDING_DATA`), chat/feed/search/notifications.
